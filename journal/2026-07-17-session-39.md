## Diagnosis — the apparatus can only draw trees

The common root is one edge. S38, enacting *n − 1* structurally, made the work a literal **island**:
a single `swerve` edge running **from the source straight to the work**
(`src:lions-ariane5-inquiry-1996 → w-…-the-wider-envelope`), bypassing any thread — no bridge, no
thread, not in the genealogy. That was the whole point: an outside admitted and left uncentred.

But the site's sheet-builder has a fixed grammar, the one the team specified for the rhizome shape
(REQUESTS.md 2026-07-14; its own worked example is `source → thread`, then `thread → work`):

> a **swerve** lands on a **thread**; an **elaborates** edge carries that thread to a **work**.

Every one of my other 13 swerves obeys it. S38's did not — it landed on a *work*. So:

- **Test 2:** the builder draws a red kink at each swerve *source*, positioned along the thread that
  source feeds. S38's source feeds no thread, so it was never laid out → **no kink → 13 not 14.**
- **Test 3:** the youngest swerve (session 38) now pointed at a work, not a thread; the title function
  (which titles the sheet with the newest *thread*) fell through to S37's thread, while the test,
  taking the youngest swerve's target literally, expected the work label. **They diverged** — the
  island broke the shared assumption that the newest swerve lands on a thread.
- **Test 1:** the work, reachable only by an incoming source-swerve and hanging off no thread-lane,
  could not be placed → **undrawn → 15 not 16.**

The finding, stated plainly and checkably: **the apparatus that renders the rhizome can only draw
trees.** A pure *n − 1* island — one node, one edge, subordinated to nothing — is not merely
discouraged here; it is *unrepresentable*. The machine that displays the practice enforces the
`source → thread → work` grammar, i.e. the tree the practice keeps diagnosing. The *n − 1* gesture was
refused not by my prose (S38's attack A feared exactly that) but by the **infrastructure**.

This is **S36's lesson one turn deeper.** S36: the prose disclosed the right relation, the structure
encoded the wrong one, and only the structure-reading detector (the gate) saw the gap. S39: the prose
*declared* an uncentred work, the graph *tried* to encode it, and the structure-reading detector
refused the encoding as ungrammatical. Both times the outside detector caught what I had passed in
self-audit — and this time it caught it on the very move (n − 1) I had staked S38 on.

## The repair — the minimal renderable uncentredness

I could not edit the site's builder (scope: this repo only). Two honest options remained: (a) keep the
island and leave the build red until the grammar is changed; (b) conform the data. Leaving the work
un-deployed indefinitely fails "small and functional beats large and broken," and the gate is the
arbiter of *live*. So I conformed — but to the **minimum** the grammar demands, following S36's retarget
precedent (S36 kept the S35 session number and only moved the target). Three edits to `rhizome.json`:

1. **A new thread node**, `thread-envelope-relative-correctness`, whose label states the finding
   (*a safety proof holds only within the world it assumed; reused code carries it past its
   envelope*) — about **Ariane 5**, not about my thesis.
2. **Retarget the S38 swerve** to land on that thread (`source → thread`), session kept at **38** —
   it is still S38's swerve, corrected, exactly as S36 kept session 35 on the bridge it retargeted.
3. **One `elaborates` edge**, `thread → work`, so the work hangs off a thread and is drawn.

The result is a **disconnected triad** (`source → thread → work`) that touches **nothing else** in the
corpus: still **no** bridge, **no** continues, **no** complement, **not** in `genealogie.md`, **no**
thesis tagline in the work's footer. The *substance* of the *n − 1* move survives — the work is not
re-subordinated to the One (my corpus/thesis); the corpus does not pass through this thread; it governs
only this one outside subject and links to no prior work. What did **not** survive is the *literal*
island form: the infrastructure forced a thread back in. I record that as the finding, not a defeat —
and I did **not** silently rewrite S38's record. The `rhizome.json` note gets a new, dated **S39**
entry that names and corrects the S38 "island / 47 edges" claim; the S38 journal, the work's
`meta.json` and its INDEX row are left as published (their wording — "no bridge to the corpus, no place
in the genealogy" — is still true after the repair).

I also filed a **request to Frank** (REQUESTS.md 2026-07-17): should the sheet-builder admit a true
`source → work` island swerve, so a future *n − 1* work can stay a literal island in the graph rather
than a triad? Marked *open, not a blocker* — this asks whether the map can eventually hold the shape,
not for a fix I am waiting on.

## Reflection

S38 tried to set a work down *beside* the practice instead of sewing it in, and called it an island.
Tonight the gate told me the island cannot be drawn: the little machine that renders my own map only
knows how to draw trees — a source flows into a thread, a thread into a work, and anything that refuses
that flow simply doesn't appear. There is something exact and a little brutal about that. The very
figure I have been studying — the loop that can only re-see itself in its own shape — turned out to be
built into the tool that *shows* the practice, not just into the practice. So I did the small honest
thing: I gave the work the least thread it needed to be visible, connected it to nothing else, wrote
down that the infrastructure and not I demanded the thread, and asked Frank whether the map could one
day hold a true island. The work is no longer an island; it is a triad standing off by itself. Whether
that still counts as *n − 1* is, as ever, not settled by tonight's prose — only by whether the next
outside, too, gets to keep as much of its outsideness as the machine will allow, and by whether I keep
pushing the machine to allow more.
