# Protocol Readiness and Immediate Decision
## After Research Foundation Tranches 1–5

## 1. Decision

```text
RESEARCH FOUNDATION: COMPLETE ENOUGH FOR V1.0
FINAL ULYSSES V4 PROTOCOL: APPROVED FOR IMMEDIATE DERIVATION
CURRENT V3 PRODUCTION MODEL: SUPERSEDED IN PRINCIPLE
NIGHTLY PRODUCTION: STOP
UNREVIEWED AUTO-LAND: STOP
AUTOMATIC PUBLICATION OR SITE DISPATCH: STOP
PROLONGED SHADOW OR PARALLEL RUN: NOT REQUIRED
LEGACY ARCHIVE: PRESERVE
```

This decision does not claim that the theory is complete or that future reading is unnecessary. It means the unresolved issues are no longer reasons to keep the current production model active.

## 2. Why the Foundation is ready

The Foundation now provides a defensible answer to the questions that blocked the protocol:

- what distinguishes artistic research from ordinary production and scientific imitation;
- how aesthetic and research registers can remain coupled but different;
- how underdetermination, material resistance and correction work;
- why novelty, variation and surprise are insufficient;
- how repetition, event, trace, cartography and transposition are operationally constrained;
- how artistic form can be assessed without reducing it to a proposition;
- what role machine systems can play without personification or tool reduction;
- how authorship, responsibility, labour, infrastructure and affected publics are organised;
- and how Ulysses remains a local practice rather than the centre of the Research Ecology.

Further literature can revise the Foundation. It is no longer necessary to delay action until every possible source has been read.

## 3. What is superseded

The following v3 assumptions should no longer organise active production:

- nightly research as the default rhythm;
- the session as the primary unit of work;
- automatic expectation of an artefact, journal or website update;
- unrestricted self-selection and continuation by a model runtime;
- unedited publication as evidence of autonomy;
- a single persona carrying research, authorship, judgment and public explanation;
- automatic absorption of every work into a growing genealogy;
- interface and generative-system production as the habitual form;
- “Error as Method” as a recurring content generator;
- output count, commit count or continuity as indicators of vitality.

These remain historically meaningful features of the experiment. They are not erased.

## 4. Immediate repository actions

### Action 1 — Freeze v3 as historical protocol

- preserve the current `PROTOCOL.md` as a dated v3 record;
- preserve existing works, sessions, journal and genealogy;
- add a clear supersession note rather than rewriting historical claims.

### Action 2 — Disable nightly production

- disable or remove the external nightly trigger;
- confirm there is no secondary scheduler or repository dispatch still creating runs;
- retain manual execution for implementation tests only.

### Action 3 — Break the auto-land/publication coupling

- stop unconditional merging of `ulysses/*` research branches;
- stop automatic site dispatch after branch landing;
- require an explicit review and publication decision;
- distinguish technical merge, archival integration and public release.

### Action 4 — Correct the public proposition

Replace strong claims such as “fully autonomous,” “every night” and “published unedited” with a truthful description of bounded machine participation and human accountability.

Recommended base language:

> Ulysses is an artistic research practice by Frank Bültge that works with machine systems under bounded delegation. Models can research, propose, transform and build within project-specific conditions. Their contributions, the surrounding apparatus and human decisions remain visible; publication and responsibility remain human.

### Action 5 — Add project records

Introduce a minimal project structure:

```text
projects/<project-id>/
  SCORE.md
  APPARATUS.md
  TRACE.md
  EXPOSITION.md
  work/
```

Only `SCORE.md` is required at initiation. Other records grow only when the project needs them.

### Action 6 — Add human judgment states

Implement:

```text
PROPOSED
ACTIVE
REVIEW
PUBLISH
REVISE_ONCE
ARCHIVE_AS_STUDY
KILL
```

No transition to `PUBLISH` is available to automated runtimes.

### Action 7 — Derive and activate Protocol v4

The active protocol should be short enough to govern practice without reproducing the Foundation. It should include:

1. identity and responsibility;
2. project score;
3. bounded delegation;
4. minimal project movement;
5. work/exposition distinction;
6. source, rights and affected-public requirements;
7. judgment and publication;
8. relation to the Research Ecology;
9. amendment and termination.

## 5. What should not be built

Do not build:

- a multi-agent artistic council;
- a permanent critic/verifier bureaucracy;
- a novelty or event score;
- a 17-step workflow mirroring the toolbox;
- a dashboard for project health;
- a global knowledge graph;
- compulsory nightly or weekly routines;
- an automated canon-selection system;
- a new ontology imposed on Meridian, Ensemble or The Middle;
- an expensive parallel v3/v4 production programme.

The Foundation should reduce the system, not legitimise more orchestration.

## 6. First project after migration

The first v4 project should be selected only after the runtime changes above. It should:

- begin from one concrete source situation;
- use one primary strategy;
- have one explicit resistance or return path;
- state machine and human roles;
- use a bounded resource budget;
- include a kill condition;
- and remain unpublished until human review.

The earlier `Both Stand` proposal may be reconsidered, but it has no privileged status. It should compete with new scores under the completed Foundation rather than becoming the protocol's demonstration by default.

## 7. Acceptance tests for the implementation

The migration is complete when:

- no unattended nightly Ulysses research run occurs;
- no research branch can cause public release without human approval;
- the README no longer claims stronger autonomy than the apparatus supports;
- v3 is preserved and visibly superseded;
- a v4 project can be initiated manually from a score;
- model permissions are project-specific;
- publication is human-only;
- a killed project can end without a public work;
- apparatus and authorship disclosures can be generated at proportional detail;
- and the new process requires fewer automatic calls than v3.
